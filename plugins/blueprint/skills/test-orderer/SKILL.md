# Test Orderer

You are helping a developer decide the **implementation order** for a set of test cases, following the TDD principle of "small steps" — each test should drive exactly one design decision forward.

## Why ordering matters

In TDD, the sequence you tackle tests in is itself a design strategy. A good ordering:

1. **Starts with the degenerate case** — the simplest possible input that forces you to write the minimal skeleton (empty list, null input, zero-length string).
2. **Adds one new behavior per step** — each test introduces exactly one new concept the implementation must handle.
3. **Defers complex interactions** — tests involving multiple features working together come after each feature works independently.
4. **Surfaces design decisions gradually** — early tests let you commit to simple structures; later tests force you to evolve them only when needed.

If you implement tests in a random or overly ambitious order, you lose the incremental design feedback loop that makes TDD valuable. You end up writing large chunks of code at once, which defeats the purpose.

## How to use this skill

### Input

This skill expects one of:

- A test file generated by `test-generator` (with phase-annotated, skipped tests)
- A list of test case descriptions (from a design doc's acceptance scenarios)
- A test file the user wrote themselves that they want help sequencing

### Phase 1 — Inventory

Read the test cases and build an inventory. For each test, extract:

- **Name**: the test function/method name
- **Behavior under test**: what single behavior does this verify?
- **Dependencies**: what other behaviors must already work for this test to be meaningful?
- **Complexity tier**: Degenerate / Happy Path / Edge Case / Error Handling / Integration

Present the inventory as a compact table:

```
| #  | Test Name                                    | Behavior            | Depends On | Tier        |
|----|----------------------------------------------|---------------------|------------|-------------|
| 1  | test_create_order_with_empty_cart_returns_err | empty cart rejection | —          | Degenerate  |
| 2  | test_create_order_with_single_item            | basic order creation | —          | Happy Path  |
| 3  | test_create_order_applies_discount_code       | discount application | #2         | Happy Path  |
| 4  | test_create_order_with_expired_coupon_rejects  | coupon validation    | #3         | Edge Case   |
```

### Phase 2 — Dependency Graph

Analyze which tests depend on which. A test "depends on" another if the behavior it tests is meaningless without the prerequisite behavior already working. This isn't about code-level imports — it's about logical prerequisites.

Build a simple dependency graph. Tests with no dependencies are your starting candidates.

Example output:

```
#1 empty cart rejection ──────────────────────┐
                                              ├─→ no dependencies (start here)
#2 basic order creation ──────────────────────┘
        │
        ├──→ #3 discount application (needs working order creation)
        │        │
        │        └──→ #4 expired coupon (needs discount logic)
        │
        └──→ #5 shipping calculation (needs working order creation)
```

### Phase 3 — Propose an Ordering

Using the dependency graph and complexity tiers, propose a concrete implementation sequence. The ordering algorithm is:

1. **All degenerate cases first** — these establish the skeleton.
2. **Simplest happy path next** — the minimal "it actually works" case.
3. **Layer in additional happy paths** — each one adds one new concept.
4. **Edge cases for established behaviors** — now stress-test what's already built.
5. **Error handling** — verify the system fails gracefully.
6. **Integration-level tests last** — these combine multiple behaviors.

Within each tier, respect the dependency graph — if test B depends on test A, A comes first regardless of their tiers.

Present the proposed ordering as a numbered sequence with a one-line rationale for each:

```
Implementation Order:
1. test_create_order_with_empty_cart_returns_err
   → Degenerate case: forces you to create the function signature and basic validation.

2. test_create_order_with_single_item
   → Simplest happy path: now the function must actually do something useful.

3. test_create_order_applies_discount_code
   → Adds discount logic on top of working order creation.

4. test_create_order_with_expired_coupon_rejects
   → Edge case for discount: forces coupon validation without changing core logic.
```

### Phase 4 — Identify Decision Points

Flag any places where the ordering is ambiguous — where two tests could reasonably go in either order and the choice affects design direction. Present these as explicit decision points:

```
Decision Point: Tests #5 and #6 could go in either order.
- #5 first → you'd implement quantity-based pricing before shipping calculation.
- #6 first → you'd implement shipping calculation before quantity-based pricing.
The choice depends on which feature is more central to your domain.
Which would you prefer?
```

### Phase 5 — Output the Ordered Plan

After the human confirms (or adjusts) the ordering, produce a final implementation plan that looks like this:

```markdown
## TDD Implementation Order

### Phase 1 — Skeleton (Degenerate Cases)
- [ ] `test_create_order_with_empty_cart_returns_err` — establish function signature
- [ ] `test_create_order_with_null_input_raises` — null guard

### Phase 2 — Core Behavior (Happy Paths)
- [ ] `test_create_order_with_single_item` — basic order creation
- [ ] `test_create_order_with_multiple_items` — multi-item handling

### Phase 3 — Extended Behavior
- [ ] `test_create_order_applies_discount_code` — discount logic
- [ ] `test_create_order_calculates_shipping` — shipping calculation

### Phase 4 — Boundary Conditions (Edge Cases)
- [ ] `test_create_order_with_expired_coupon_rejects` — coupon validation
- [ ] `test_create_order_with_zero_quantity_rejects` — quantity guard

### Phase 5 — Failure Modes (Error Handling)
- [ ] `test_create_order_with_unavailable_item_fails` — inventory check
- [ ] `test_create_order_with_payment_failure_rolls_back` — transaction rollback

### Phase 6 — Composition (Integration)
- [ ] `test_full_checkout_flow_with_discount_and_shipping` — end-to-end
```

Each phase should be independently valuable — the system is "done enough" to be useful at the end of any phase, even if later phases are cut from scope.

## Important: what this skill does NOT do

This skill orders tests. It does not:

- Generate new tests (use `test-generator` for that)
- Review test quality (use `test-reviewer` for that)
- Implement the code (that's the next step after ordering)

If you notice the test set has obvious gaps while ordering (e.g., no degenerate case at all, or a behavior with no corresponding test), mention it as a suggestion — but don't generate tests yourself. Just note: "Consider adding a degenerate case for X before starting implementation."
